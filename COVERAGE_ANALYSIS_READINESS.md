# Coverage Analysis Readiness Report

Research investigation into DOCSTAR codebase readiness for a Scorecard phase with coverage analysis.

---

## 1. Shadow Doc Format and Content

### Output Format

Shadow docs are plain markdown files with a structured header. They are generated by the `generate_file_shadow_doc_async()` function in `src/docstar/shadow.py:154-226`.

**File-level shadow doc structure** (assembled at `shadow.py:60-64`):

```markdown
# <relative_path>
@source-hash: <sha256_hex>
@generated: <ISO8601_timestamp>

<body — semantically dense markdown summary>
```

**Directory-level roll-up structure** (assembled at `shadow.py:67-71`):

```markdown
# <relative_path>/
@children-hash: <merkle_hash_hex>
@generated: <ISO8601_timestamp>

<body — synthesized summary of all child file/directory summaries>
```

### Shadow Doc Body Content

The body is free-form markdown generated by the LLM with these instructions (`shadow.py:167-185`, system prompt):

> Shadow docs are semantically dense summaries that help AI agents quickly understand code.

The tool description (`tools.py:11-22`) specifies the body should include:

- Primary purpose and responsibility of the file
- Key classes, functions, and their roles (with line numbers)
- Important dependencies and relationships
- Notable patterns or architectural decisions
- Any critical invariants or constraints

The user prompt (`shadow.py:195-196`) also instructs:

> Include line number references for key elements (e.g., "MyClass (L15-45)").

### Do Shadow Docs Contain a Summary/Abstract?

**No.** Shadow docs do **not** have a fixed-format abstract, summary field, or topic list. The entire body is a free-form markdown summary. There is no structured metadata beyond the three header lines (`@source-hash`, `@generated`, and the `# path` title).

The body content is dense and informative, but its structure varies per file — sometimes it starts with a one-line purpose statement, sometimes it dives directly into component descriptions. There is no guaranteed "topic signature" section.

### Metadata Stored Alongside Shadow Docs

Three sidecar artifact types are written during shadow generation:

**1. Findings JSON** — `shadow.py:321-341`, stored at `.docstar/findings/<path>.findings.json`:
```json
{
  "source": "src/docstar/config.py",
  "source_hash": "abc123...",
  "generated": "2025-02-22T12:10:00Z",
  "findings": [
    {
      "category": "stale_comment",
      "line_start": 42,
      "line_end": 42,
      "severity": "warning",
      "description": "...",
      "suggestion": "..."
    }
  ]
}
```

**2. Symbols JSON** — `shadow.py:343-353`, stored at `.docstar/symbols/<path>.symbols.json`:
```json
{
  "source": "src/docstar/config.py",
  "source_hash": "abc123...",
  "generated": "2025-02-22T12:10:00Z",
  "symbols": [
    {
      "name": "Config",
      "kind": "class",
      "line_start": 112,
      "line_end": 232
    }
  ]
}
```

**3. Token cache** — `.docstar/token-cache.json` (used by `stats.py`, not per-file).

### Actual `.docstar/shadow/` Output Files

**None exist in the repository.** The `.docstar/` directory is listed in `.gitignore` (via `DEFAULT_IGNORE_PATTERNS` at `config.py:26`), so no generated output is committed. All shadow docs, findings, and symbols are runtime-generated artifacts.

## 2. Documentation Analysis Output (Phase 2)

### Data Structures

Phase 2 is implemented in `src/docstar/debris.py`. Two key dataclasses:

**DocFinding** (`debris.py:21-30`):
```python
@dataclass
class DocFinding:
    category: str       # stale_content, incorrect_content, obsolete_reference, misleading_claim
    severity: str       # error, warning
    description: str
    shadow_ref: str     # source path of evidencing shadow doc
    evidence: str       # quote from shadow doc
    remediation: str
```

**DocAnalysisResult** (`debris.py:33-46`):
```python
@dataclass
class DocAnalysisResult:
    path: Path
    classification: str  # Diataxis category
    confidence: float
    classification_reason: str
    matched_shadows: list[str] = field(default_factory=list)
    findings: list[DocFinding] = field(default_factory=list)

    @property
    def is_debris(self) -> bool:
        return self.classification == "process_artifact"
```

### Diataxis Type Classification

**Yes, Phase 2 already classifies docs by Diataxis type.** The classification is returned from the Opus LLM call and stored in `DocAnalysisResult.classification`. The enum values are defined in the tool schema (`tools.py:155-157`):

- `reference` — Precise technical information (API docs, specs, ADRs, design docs)
- `tutorial` — Learning-oriented walkthrough for beginners
- `how-to` — Task-oriented guide for specific goals
- `explanatory` — Understanding-oriented discussion of concepts
- `process_artifact` — Inherently temporary file (debris)

The classification includes a `confidence` float (0.0-1.0) and a `classification_reason` string explaining the classification.

### Topic Summary / Description of What Each Doc Covers

**Partially.** Phase 2 does **not** produce an explicit topic summary for each doc. However, it does produce:

1. `classification_reason` — a brief explanation of why the doc was classified as it was. This is a natural language string, not a structured topic list.
2. `matched_shadows` — a list of source file paths that the doc was matched against (via Tier 1 explicit refs + Tier 2 Haiku topic matching). This tells us *which code* a doc covers, but not *what concepts/topics* it addresses.

The Haiku topic matching step (`debris.py:207-270`) matches docs to relevant source directories, but it only returns directory paths — no topic descriptions or concept labels.

### Severity Classification Schema

Two severity levels used consistently across all phases:

- `error` — blocking issue (causes audit failure)
- `warning` — non-blocking issue

For documentation findings specifically, the categories are (`tools.py:176-181`):

| Category | Description |
|---|---|
| `stale_content` | Information is outdated |
| `incorrect_content` | Contradicts source code |
| `obsolete_reference` | References deleted/renamed symbols |
| `misleading_claim` | Described behaviors the code doesn't implement |

Each finding also has a `confirmed` boolean. Only findings where `confirmed: true` are kept — this allows the LLM to retract findings it reconsiders mid-generation (`debris.py:380-381`).

## 3. Code Findings Output (Phase 3)

### Schema of `.docstar/findings/*.findings.json`

Written at `shadow.py:325-341`:

```json
{
  "source": "<relative_path>",
  "source_hash": "<sha256>",
  "generated": "<ISO8601>",
  "findings": [
    {
      "category": "<enum>",
      "line_start": "<int>",
      "line_end": "<int>",
      "severity": "error|warning",
      "description": "<string>",
      "suggestion": "<string|null>"
    }
  ]
}
```

### Finding Type Categories

Defined in `tools.py:38-44`:

| Category | Description |
|---|---|
| `stale_comment` | Describes behavior the code no longer exhibits |
| `misleading_docstring` | Doesn't match actual implementation |
| `commented_out_code` | Code blocks 3+ lines that agents might reference |
| `expired_todo` | TODO/FIXME with outdated context |
| `dead_code` | Unreachable branches, unused functions (within-file) |

### Severity Classification

Yes — same `error`/`warning` enum as other phases, defined in the tool schema (`tools.py:48-50`).

### Dead Code Verification (Phase 4 — optional)

When `--dead-code` is passed, `deadcode.py` produces `DeadCodeVerification` objects (`deadcode.py:43-55`):

```python
@dataclass
class DeadCodeVerification:
    source_path: str
    name: str
    kind: str           # function, class, constant, variable
    line_start: int
    line_end: int | None
    is_dead: bool
    confidence: float
    reason: str
    remediation: str
```

Note: Dead code results are **not** written to JSON files. They exist only in memory and are surfaced as `AuditIssue` objects during audit orchestration (`audit.py:187-196`).

### Do Any Existing Prompts Ask for Topic Extraction?

**No.** None of the five prompts ask the LLM to produce topic signatures, concept labels, or structured topic lists. The closest is:

- **Prompt 3 (Haiku topic matcher)** — performs semantic matching between a doc and code directories, but outputs *directory paths* not topics/concepts.
- **Prompt 4 (Opus analyzer)** — produces a `classification_reason` which loosely describes what the doc is about, but this is unstructured and varies widely.
- **Prompt 1 (shadow generation)** — the output body is a dense summary, but there's no instruction to produce a fixed-format topic list.

## 5. Gap Analysis for Coverage

### Can Topic Signatures Be Derived from Existing Shadow Doc Output Without an Additional LLM Call?

**No, not reliably.** Shadow doc bodies are free-form markdown summaries with no fixed structure. While they are dense and informative, extracting a consistent ~50-80 token "topic signature" would require either:

1. **Heuristic extraction** — Taking the first 1-2 sentences of the body. This might work for files where the LLM starts with a purpose statement, but the format is not guaranteed.
2. **An additional LLM pass** — A Haiku call to distill each shadow doc into a fixed-format signature.

The `public_symbols` array in the symbols JSON sidecar provides the *public surface* component of a topic signature (function/class names with kinds). This is structured, consistent, and already exists. But it doesn't capture *concepts* or *topics* — just symbol names.

**Recommendation:** Topic signatures for code need a new Haiku pass. However, the `public_symbols` list can serve as the "public surface" half of the signature without any additional call. Only the "key concepts/topics" half needs generation.

### Can Topic Signatures for Human Docs Be Derived from Existing Phase 2 Output?

**No.** Phase 2 produces:

- `classification` (Diataxis type) — useful for the coverage matrix column headers, but not a topic signature.
- `classification_reason` — unstructured, variable-length, not a topic list.
- `matched_shadows` — tells us which source files the doc relates to, which is useful for the *mapping* step but not for generating the doc's own topic signature.

A new Haiku pass would be needed to generate topic signatures for human docs.

**However**, there is a potential optimization: the Haiku topic matching step (Prompt 3) already reads the full doc content. If topic signature generation were added to that prompt, the two tasks could be combined into a single Haiku call per doc.

### What's Missing?

#### New Data Structures Needed

1. **TopicSignature** — A new dataclass for the fixed-format summary of each document/shadow doc:
   ```python
   @dataclass
   class TopicSignature:
       path: str                    # relative path
       kind: str                    # "source" or "doc"
       topics: list[str]            # key concepts/topics covered
       public_surface: list[str]    # for source: function/class names; for docs: sections/headings
       diataxis_type: str | None    # for docs: classification from Phase 2
   ```

2. **CoverageEntry** — Mapping from codebase concept to covering docs:
   ```python
   @dataclass
   class CoverageEntry:
       concept: str                 # codebase concept
       source_path: str             # where the concept lives in code
       covering_docs: list[dict]    # [{path, diataxis_type, relevance}]
   ```

3. **Scorecard** — Aggregated metrics:
   ```python
   @dataclass
   class Scorecard:
       coverage_matrix: list[CoverageEntry]
       coverage_pct: float          # % of codebase concepts with at least one doc
       dead_docs: list[str]         # docs classified as process_artifact
       accuracy_errors_per_doc: dict[str, int]
       hygiene_warnings_per_file: dict[str, int]
   ```

#### New Prompts Needed

1. **Topic signature extraction for shadow docs** — A Haiku pass that reads each shadow doc body and produces a structured ~50-80 token signature. Could potentially be added to the existing shadow generation prompt (Prompt 1), but that would increase output token cost for every shadow generation run, even when not auditing. Better as a separate lightweight pass.

2. **Topic signature extraction for human docs** — A Haiku pass per doc. Could be combined with the existing Prompt 3 (topic matching) to avoid an extra call.

3. **Coverage mapping** — A single Opus call that receives all topic signatures and produces the coverage matrix. This is entirely new.

#### New Processing Needed

1. **Topic signature generation phase** — Between current Phase 2 and the new Scorecard phase. Reads shadow docs and human docs, produces signatures.
2. **Coverage matrix generation** — Single Opus call with all signatures.
3. **Scorecard tabulation** — Pure Python counting from coverage matrix + existing phase outputs.

### What's the Current Finding Aggregation Story?

**There is existing aggregation, but it's simple.** The `run_audit()` function in `audit.py:80-198` is the sole aggregation point. It:

1. Collects `AuditIssue` objects from each phase into a flat list
2. Distinguishes `error` vs `warning` severity
3. Produces two output formats: markdown report (`format_audit_report`) and JSON (`format_audit_json`)

The aggregation is purely additive — there is no:
- Per-file metric rollup
- Cross-phase correlation
- Coverage analysis
- Summary statistics beyond error/warning counts

The Scorecard phase would be building the metrics rollup from scratch, but it can read all existing outputs (findings JSON, symbols JSON, shadow docs, `DocAnalysisResult` objects).

## 6. Data Flow

### Phase-by-Phase Data Flow

```
Phase 1: Shadow Generation (shadow.py)
  Reads:  Source files on disk
  Writes: .docstar/shadow/<path>.shadow.md      (per file)
          .docstar/shadow/<dir>/_directory.shadow.md  (per directory)
          .docstar/findings/<path>.findings.json  (per file)
          .docstar/symbols/<path>.symbols.json    (per file)
  LLM:    Sonnet (per file + per directory)

Phase 2: Documentation Analysis (debris.py)
  Reads:  Human-written doc files (discovered via config.is_doc_candidate)
          .docstar/shadow/_directory.shadow.md files (for topic matching)
          .docstar/shadow/<path>.shadow.md files (for accuracy validation)
          .docstar/rules (optional override rules)
  Writes: Nothing to disk (returns list[DocAnalysisResult] in memory)
  LLM:    Haiku (topic matching, per doc) + Opus (classification+validation, per doc)

Phase 3: Code Findings Surfacing (audit.py:149-177)
  Reads:  .docstar/findings/*.findings.json
  Writes: Nothing (creates AuditIssue objects in memory)
  LLM:    None (pure file I/O)

Phase 4: Dead Code Detection (deadcode.py) — optional
  Reads:  .docstar/symbols/*.symbols.json
          All repo files (for grep-based reference scanning)
          .docstar/shadow/<path>.shadow.md (for LLM verification context)
          Source files (for LLM verification context)
  Writes: Nothing to disk (returns list[DeadCodeVerification] in memory)
  LLM:    Sonnet (per low-reference candidate)

Audit Orchestration (audit.py:80-198)
  Reads:  All phase outputs (in memory or from disk)
  Writes: Nothing (returns AuditResult with flat list of AuditIssue)
  Output: Markdown report or JSON (to stdout)
```

### Where Would a Phase 5 (Scorecard) Read Its Inputs?

A Scorecard phase would need to read from:

| Input | Source | Format |
|---|---|---|
| Shadow doc bodies | `.docstar/shadow/**/*.shadow.md` | Markdown with header |
| Public symbols | `.docstar/symbols/**/*.symbols.json` | JSON |
| Code findings | `.docstar/findings/**/*.findings.json` | JSON |
| Doc analysis results | In-memory `list[DocAnalysisResult]` from Phase 2 | Python objects |
| Dead code verifications | In-memory `list[DeadCodeVerification]` from Phase 4 | Python objects |

**Key architectural issue:** Phases 2 and 4 produce in-memory results only — they do not persist to disk. For the Scorecard to be a separate phase (rather than appended to `run_audit()`), either:

1. Phase 2 and 4 results would need to be serialized to disk (e.g., `.docstar/analysis/*.json`), or
2. The Scorecard logic would need to run inside `run_audit()` with access to the in-memory results.

Option 2 is simpler and is the current pattern — `run_audit()` already receives all phase results in memory.

### Is There a Unified Findings Format?

**Partially.** The `AuditIssue` dataclass (`audit.py:15-25`) serves as the unified format:

```python
@dataclass
class AuditIssue:
    path: Path
    severity: str       # "error" or "warning"
    category: str       # namespace varies by phase
    message: str
    remediation: str
    line_start: int | None = None
    line_end: int | None = None
```

But the category namespace is ad-hoc across phases:

| Phase | Category prefix | Examples |
|---|---|---|
| Phase 1 (shadow check) | `*_shadow` | `stale_shadow`, `missing_shadow` |
| Phase 2 (doc analysis) | `debris` or `doc_*` | `debris`, `doc_stale_content`, `doc_incorrect_content` |
| Phase 3 (code findings) | raw category | `stale_comment`, `misleading_docstring`, `commented_out_code` |
| Phase 4 (dead code) | `cross_file_dead_code` | `cross_file_dead_code` |

There is no formal enum or registry of all category values. The category strings are constructed inline in `audit.py`.

## Summary of Key Findings

### What Already Exists (Reusable for Coverage)

1. **Diataxis classification for human docs** — `DocAnalysisResult.classification` provides the type axis of the coverage matrix.
2. **Public symbols per source file** — `.docstar/symbols/*.symbols.json` provides the "public surface" component of topic signatures.
3. **Doc-to-code matching** — `DocAnalysisResult.matched_shadows` provides which source files each doc covers (via Tier 1 regex + Tier 2 Haiku matching).
4. **Unified issue aggregation** — `AuditIssue` and `AuditResult` provide the framework for rolling up findings, though currently only counting errors vs warnings.
5. **All code findings on disk** — `.docstar/findings/` provides per-file code debris counts.

### What Needs to Be Built

1. **Topic signature generation** — New Haiku pass to distill shadow docs and human docs into fixed-format ~50-80 token signatures.
2. **Coverage mapping prompt** — New Opus call receiving all signatures and producing a coverage matrix.
3. **Scorecard data structures** — `TopicSignature`, `CoverageEntry`, `Scorecard` dataclasses.
4. **Metric tabulation** — Pure Python counting for dead docs, accuracy errors/doc, hygiene warnings/file from existing data.
5. **Scorecard reporting** — Extension to `format_audit_report()`/`format_audit_json()` for the new metrics.

### Potential Optimization: Reduce New LLM Calls

The existing Phase 2 Haiku call (Prompt 3) already reads full doc content and directory summaries. The topic signature for human docs could be generated as an additional output field in `match_doc_topics`, avoiding a separate call. This would require:

- Adding a `topic_signature` field to the `MATCH_DOC_TOPICS_TOOL` schema
- Updating `_MATCH_SYSTEM_PROMPT` to also produce topic keywords
- Updating `_match_topics_async()` to extract the signature

Similarly, the shadow generation prompt (Prompt 1) already produces `public_symbols`. A `topic_summary` field could be added to `submit_shadow_doc` to get topic signatures during shadow generation itself, but this adds output tokens to every shadow generation run (not just audits).
